# [infra-7-2] 該看哪些指標？伺服器的生命徵象

> **本章目標**：建立「看伺服器健康該看什麼」的判斷力——CPU、記憶體、磁碟、網路四大生命徵象，以及它們各自「多少算危險」。

## 你會學到

- 伺服器的四大核心指標：CPU、記憶體、磁碟、網路
- 每個指標看什麼、超過多少要警覺
- 「飽和度」這個關鍵概念
- 主機指標 vs 服務指標的差別

## 概念說明

### 日誌看「發生了什麼事」，指標看「現在好不好」

上一章的日誌，回答的是「**發生了什麼事**」（離散的事件）。這一章的**指標（metrics）**，回答的是「**現在狀況好不好**」（持續變化的數值）。

兩者互補，就像看病：日誌是「病歷上記的事件」，指標是「即時的生命徵象——體溫、心跳、血壓」。一個稱職的 infra 工程師，要能一眼讀懂機器的「生命徵象」，在它病倒之前就察覺異常。

伺服器的「生命徵象」主要是四個：**CPU、記憶體、磁碟、網路**。

---

### 四大生命徵象

```mermaid
graph TB
    CPU["CPU<br/>運算力夠不夠用"]
    MEM["記憶體 RAM<br/>暫存空間夠不夠"]
    DISK["磁碟<br/>儲存空間 + 讀寫速度"]
    NET["網路<br/>流量與連線狀況"]
```

這四個你在 Part 2 都摸過了，這裡把它們組成一套「健康判讀」的框架：

| 指標 | 看什麼 | 危險訊號 | 太高代表 | 複習 |
|------|--------|---------|---------|------|
| **CPU** | 使用率、load average | load 持續 > 核心數 | 算不過來，請求開始變慢 | Part 2-3 |
| **記憶體** | 已用 / 可用、swap 使用 | 可用記憶體趨近 0、瘋狂用 swap | 快撐爆，可能觸發 OOM 把程式殺掉 | Part 2-3 |
| **磁碟** | 使用率、I/O 等待 | `Use%` > 80~90% | 快寫不下，服務開始報錯 | Part 2-4 |
| **網路** | 流量、連線數、錯誤 | 頻寬打滿、連線爆量 | 塞車，回應變慢或拒絕連線 | Part 3 |

### 一個關鍵概念：飽和度（Saturation）

新手常只看「使用率」，但更該關心的是**飽和度**——「**這個資源還有多少餘裕，還是已經滿到開始排隊了**」。

用高速公路類比：

- **使用率 70%**：車不少，但還順暢。健康。
- **使用率 95%**：快塞滿了，稍有風吹草動就堵死。危險邊緣。
- **飽和（saturation）**：已經塞爆、車子在排隊（例如 CPU 的 load average 超過核心數、記憶體開始用 swap）。**這才是真正要警報的狀態**。

所以判讀指標時，不是只看「現在用多少」，而是看「**還剩多少餘裕、有沒有開始排隊**」。一個 CPU 使用率 100% 但工作都做完了的瞬間沒關係；持續飽和、請求在排隊，才是問題。

---

### 主機指標 vs 服務指標

指標還分兩個層次，都要看：

- **主機指標（host metrics）**：整台機器的 CPU、記憶體、磁碟、網路。回答「**這台機器健康嗎**」。
- **服務指標（service metrics）**：你的應用本身的數字——每秒處理幾個請求、回應要多久、錯誤率多少。回答「**我的服務對使用者好不好**」。

主機很健康，不代表服務沒問題（可能程式有 bug 一直回錯誤）；反過來也是。兩個層次要一起看，才有完整圖像。

> 服務層次的指標，有一套很經典的框架叫「**四個黃金訊號**」（延遲、流量、錯誤、飽和度），那是 SRE 的核心方法論。想深入 → 參見 **SRE 課程**：`lessons/sre/課程大綱.md`（Part 3 觀測性）。這門 infra 課我們先把「主機指標」打穩。

## 程式碼範例

這些都是「手動快速看一眼」的工具（你 Part 2、3 大多用過）。下一章會把它們升級成「自動持續監控」。

**CPU**——看 load average 與使用率：

```bash
uptime          # load average 三個數字
top             # 即時，看哪個程式吃 CPU
nproc           # 你有幾核（判斷 load 高低的基準）
```

**記憶體**——看可用量與 swap：

```bash
free -h
```

重點看 `available`（真正還能用的）和 `Swap` 那行（如果 swap 用很多，代表 RAM 不夠了）。

**磁碟**——空間與 I/O：

```bash
df -h                    # 空間使用率
iostat -x 2 3            # I/O 狀況（可能要先 apt install sysstat）
```

`iostat` 看磁碟讀寫忙不忙，`%util` 接近 100% 代表磁碟 I/O 飽和。

**網路**——連線與流量：

```bash
ss -s                    # 連線數量統計
ss -tlnp                 # 哪些 port 在聽（Part 3-1 用過）
```

### 把「四大生命徵象」做成一鍵巡檢

你已經會寫腳本了（Part 6-1）！把這些組合成一個巡檢腳本，就是一份「手動健康檢查」。但手動看有個根本問題：**你不可能 24 小時盯著終端機**。半夜三點機器飽和了，你在睡覺，根本不知道。

這正是下一章要解決的——**用 Prometheus + Grafana 把這些指標自動收集、畫成圖、並在出事時主動通知你**。

## 小練習

### 練習 1：判讀生命徵象

下面的狀況，你會擔心哪些？為什麼？

1. CPU 使用率 100%，但 load average 0.5（機器有 4 核）
2. 記憶體 available 剩 50MB，swap 用了 2GB
3. 磁碟 `Use%` 是 92%
4. load average 是 8.0（機器有 2 核）

> 提示：用「飽和度」的角度想——哪些是「還有餘裕」，哪些是「已經在排隊」。

---

### 練習 2：分清兩個層次

判斷下面是「主機指標」還是「服務指標」：

1. 這台機器的 CPU 用了 60%
2. 我的 API 平均回應時間 200ms
3. 磁碟還剩 10GB
4. 網站這分鐘有 3% 的請求回傳 500 錯誤

---

### 練習 3：對你的伺服器做一次完整體檢

用本章的指令，把你伺服器的四大生命徵象各看一遍，判斷它現在健康嗎。把結果記下來——下一章裝好監控後，你會看到同樣的數字被自動畫成漂亮的即時圖表。
