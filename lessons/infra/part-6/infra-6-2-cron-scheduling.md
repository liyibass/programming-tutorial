## 概念說明

### 痛點：有些事得「定時做」，但人會忘記

很多維運工作不是做一次就好，而是要**規律地重複**：

- 每天凌晨備份資料庫
- 每小時清理暫存檔
- 每週產生一份報表
- 每分鐘檢查服務還活著嗎

這些事如果靠人「記得去做」，遲早會忘、會漏。Part 4-4 我們也看過——certbot 的憑證續期就是「定時自動」做的，沒人會記得每 90 天手動續一次。

**cron** 就是解決這件事的工具：你告訴它「**什麼時間、執行什麼**」，它就會像鬧鐘一樣，時間到自動幫你跑，不用你盯著。

用類比：cron 是伺服器的**行事曆鬧鐘**——你把「每天早上 3 點：備份」寫進去，它每天 3 點準時響、準時做，風雨無阻。

---

### cron 的時間格式：五個欄位

cron 的設定（叫 **crontab**，cron table）每一行長這樣：**五個時間欄位 + 要執行的指令**。

```
┌── 分鐘 (0-59)
│ ┌── 小時 (0-23)
│ │ ┌── 日 (1-31)
│ │ │ ┌── 月 (1-12)
│ │ │ │ ┌── 星期 (0-7，0 和 7 都是星期日)
│ │ │ │ │
* * * * *  要執行的指令
```

每個欄位填數字代表「在這個時間點」，填 `*` 代表「每一個都算」（不限制）。看幾個例子最快懂：

| 設定 | 意思 |
|------|------|
| `0 3 * * *` | **每天**凌晨 3:00 |
| `*/10 * * * *` | **每 10 分鐘**（`*/10` = 每隔 10） |
| `0 * * * *` | **每小時**整點（第 0 分） |
| `0 0 * * 0` | **每週日**午夜 |
| `30 2 1 * *` | **每月 1 號**的 2:30 |

口訣：**從左到右是「分、時、日、月、週」**。一開始會不習慣，多看幾個例子、或用線上的 cron 產生器輔助就熟了。

## 程式碼範例

### 編輯你的 crontab

每個使用者有自己的 crontab。編輯它：

```bash
crontab -e
```

第一次會問你選編輯器，選 `nano` 最簡單。打開後，在最下面加一行。例如「每天凌晨 3 點，執行 Part 6-1 的健康檢查腳本」：

```
0 3 * * * /home/deploy/infra-practice/healthcheck.sh
```

> 重要：cron 裡**一定要用完整路徑**（`/home/deploy/...`，不能用 `~` 或相對路徑），因為 cron 執行時的環境跟你登入時不一樣，它不知道你的「家」在哪。

存檔離開，這條排程就生效了。

---

### 把輸出存進日誌

腳本跑的時候會有輸出，但 cron 在背景跑，你看不到。最好把輸出「導向」到一個日誌檔，之後出問題能查：

```
0 3 * * * /home/deploy/infra-practice/healthcheck.sh >> /home/deploy/logs/health.log 2>&1
```

拆解結尾那串：

- `>> /home/deploy/logs/health.log`：把輸出**附加**到這個日誌檔（`>>` 是附加，`>` 是覆蓋）。
- `2>&1`：把「錯誤訊息」也一起導到同一個檔（`2` 是錯誤輸出、`1` 是標準輸出，這串意思是「錯誤也跟著正常輸出走」）。

這樣每次執行的結果都會留下記錄，呼應 Part 1-1 說的「能查線索」的維運習慣。

---

### 查看與管理排程

列出你目前所有的排程：

```bash
crontab -l
```

`-l` 是 list。要修改就再 `crontab -e`，把那行刪掉或改掉即可。

---

### cron 與 systemd timer

你在 Part 4-4 看過 `certbot.timer`——那是 **systemd timer**，是另一種排程方式。它和 cron 做的事類似，但和 systemd 整合更好（能用 `journalctl` 看日誌、有更靈活的觸發條件）。

簡單分辨：

| | cron | systemd timer |
|---|------|--------------|
| 設定方式 | `crontab -e`，一行搞定 | 寫 `.timer` + `.service` 兩個檔 |
| 適合 | 簡單的定時任務 | 需要和服務整合、複雜觸發 |
| 上手難度 | 低 | 中 |

日常簡單排程用 cron 就很夠了；當你需要更進階的控制，再認識 systemd timer。兩者並存，不衝突。

> WSL 注意：cron 在 WSL 上可用，但要確保 cron 服務有在跑（`sudo service cron start`，或 Part 0 啟用 systemd 後用 `systemctl`）。WSL 沒開著的時候排程不會執行——所以「每天凌晨自動跑」這類任務，正式環境還是要放在一直開機的雲端伺服器上。

> 提示：每 30 分鐘是 `*/30 * * * *`。
