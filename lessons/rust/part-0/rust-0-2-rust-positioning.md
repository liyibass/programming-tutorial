# [rust-0-2] Rust 站在哪：和 C/C++、Go、TypeScript/C# 的定位比較

> **本章目標**：把 Rust 放進「程式語言地圖」裡，看清楚它和你可能聽過的其他語言各自擅長什麼、犧牲什麼，幫你判斷「什麼時候該選 Rust」。

## 你會學到

- 用兩個軸（執行速度、開發便利）把常見語言排排站
- Rust 和 C/C++、Go、TypeScript/C# 各自的取捨
- 「系統程式語言」vs「應用程式語言」是什麼意思
- 什麼情境適合選 Rust、什麼情境其實不必

## 概念說明

### 沒有「最好」的語言，只有「最適合某情境」的語言

每種語言都是一組**取捨**的結果。與其問「哪個語言最強」，不如問「這個工作，我願意拿什麼換什麼」。我們用兩個常見的軸來看：

- **執行速度 / 對底層的掌控**：程式跑得多快、能不能精細控制記憶體。
- **開發便利 / 上手速度**：寫起來多快、多省心、多容易學。

### 一個一個比

**Rust vs C / C++**

同樣是「系統程式語言」（能直接掌控記憶體、沒有 GC），效能同級。差別在**安全**：C/C++ 把「不要踩記憶體地雷」的責任完全交給你；Rust 用編譯器幫你檢查，擋掉一整類記憶體與並行 bug（[rust-0-1] 講過）。代價是 Rust 編譯器很嚴格，要先學會所有權。

> **一句話**：想要 C/C++ 的速度，但不想再為記憶體 bug 熬夜除錯 → Rust。

**Rust vs Go**

Go 也是為「現代後端、雲端服務」而生，而且**非常好上手**、內建好用的並行模型。但 Go **有 GC**。所以：

- 要「快速開發一個普通後端服務、團隊好招人」→ Go 往往更省事。
- 要「極致效能、可預測的延遲（不能有 GC 暫停）、或要寫系統/嵌入式」→ Rust 更合適。

**Rust vs TypeScript / C#（你在 basic、csharp 課學的）**

TypeScript、C# 是**應用程式語言**——有 GC、生態成熟、開發飛快，做網站、後端 API、企業系統都很順。它們和 Rust 不太算「競爭對手」，而是**不同層次的工具**：

| | TypeScript / C# | Rust |
|---|---|---|
| 記憶體 | GC 自動管 | 所有權，編譯期管 |
| 上手 | 快 | 慢（所有權要學） |
| 典型用途 | 網站、後端 API、企業應用 | 系統、高效能服務、嵌入式、WASM |
| 心智負擔 | 低 | 高（但換來安全與速度） |

### 「系統程式語言」到底是什麼意思

你會一直聽到「Rust 是系統程式語言」。簡單說：

- **應用程式語言**：寫「給人用的軟體」（網站、App、後端服務），通常站在 GC 與作業系統的肩膀上，不太碰底層。
- **系統程式語言**：寫「給機器/其他軟體用的底層東西」（作業系統、瀏覽器引擎、資料庫、驅動程式、嵌入式韌體），需要直接掌控記憶體與硬體、不能有額外開銷。

Rust 的野心是：**讓你能寫系統級的東西，卻不必承受 C/C++ 那種隨時出包的風險。**

## 程式碼範例

這章是概念比較，沒有要你寫程式。但給你一個「同一件事、不同語言味道」的對照感受——印出一行字：

```rust
// Rust
fn main() {
    println!("Hello");
}
```

```typescript
// TypeScript
console.log("Hello");
```

```go
// Go
package main
import "fmt"
func main() { fmt.Println("Hello") }
```

看起來都差不多對吧？**語言的差異不在「印一行字」這種小事**，而在「當程式變大、要管理大量資料與並行時，誰幫你擋住錯誤、誰跑得更快」。那才是選 Rust 或不選 Rust 的真正分水嶺。

## 小練習

1. 用「取捨」的角度，各寫一句話總結：為什麼有人選 Go 而不選 Rust？為什麼有人選 Rust 而不選 C++？
2. 假設以下三個專案，你會傾向選哪個語言、為什麼？（沒有標準答案，重點是說出理由）
   - (a) 一個公司內部用的請假系統網站
   - (b) 一個要塞進智慧手錶、電力與記憶體都很拮据的韌體
   - (c) 一個每秒要處理數十萬筆、延遲不能有突然尖刺的即時交易引擎
3. 查一個你有興趣的知名軟體（例如某瀏覽器、某資料庫），看它底層用什麼語言寫，想想為什麼。
