# [rust-7-4] 文件註解與 `cargo doc`：讓程式碼自我說明

> **本章目標**：學會寫 Rust 的文件註解，並用 `cargo doc` 一鍵產生漂亮的網頁文件——讓你（和別人）半年後還看得懂這段程式在幹嘛。

## 你會學到

- 一般註解 vs 文件註解的差別
- 用 `///` 寫文件註解
- 用 `cargo doc` 產生 HTML 文件
- 文件測試（doc test）：連範例都會被驗證

## 概念說明

### 註解是寫給未來的人看的

程式碼是給電腦執行的，但**也是給人讀的**——尤其是「半年後的你」和「接手的同事」。好的註解能省下大量「這到底在幹嘛」的考古時間。

Rust 有兩種註解：

```
// 一般註解          ── 給「正在讀這段原始碼」的人看的內部說明
/// 文件註解         ── 給「要『使用』這個函式」的人看的說明文件，
                       還能被工具抽出來變成漂亮的網頁文件
```

文件註解的特別之處：`cargo doc` 會把所有 `///` 收集起來，產生一份**專業的 HTML 文件**——就跟你在官方網站看到的標準庫文件長得一模一樣。

> 註解該寫「為什麼」而非「做什麼」（程式碼本身就說明做什麼）→ [課外讀物 E-6-5：註解的藝術](../../../課外讀物/E-6-best-practices/E-6-5-comments.md)

## 程式碼範例

### 用 /// 寫文件註解

文件註解寫在「要說明的項目正上方」，用三條斜線 `///`：

```rust
/// 計算兩個數字的和。
///
/// # 範例
///
/// ```
/// let result = my_crate::add(2, 3);
/// assert_eq!(result, 5);
/// ```
pub fn add(a: i32, b: i32) -> i32 {
    a + b
}
```

說明：

- `///` 後面寫說明文字，支援 **Markdown**（可以用標題 `#`、清單、程式碼區塊）。
- 慣例會包含「這個函式做什麼」、常見的區段如 `# 範例`、`# Panics`（什麼情況會 panic）、`# Errors`（回傳 `Err` 的情況）。
- 那段用 ` ``` ` 包起來的範例程式碼，不只是給人看——它還會被當成測試執行（見下面）！

### 產生並瀏覽文件

```bash
cargo doc --open
```

這個指令會：
1. 把你所有 `///` 註解 + 函式簽名，產生成 HTML 文件。
2. `--open` 自動在瀏覽器打開。

你會看到一份排版精美、可搜尋、能點擊跳轉的文件——而且它**永遠和程式碼同步**（因為就是從程式碼產生的）。這比「另外維護一份 Word 文件」可靠太多了，因為分開的文件總會過時。

### 文件測試：範例也會被驗證

Rust 有個很厲害的特性——**文件註解裡的程式碼範例，會被 `cargo test` 當成測試跑**（叫 doc test）：

```bash
cargo test
```

它除了跑你 [rust-7-3] 寫的測試，**還會把每段 `///` 裡的 ` ``` ` 範例編譯並執行**，確認它們真的能跑、結果正確。

這解決了一個常見痛點：**文件裡的範例程式碼過時、根本跑不動**。在 Rust，只要範例壞了，`cargo test` 就會失敗，逼你更新——所以 Rust 的文件範例幾乎總是可信的。文件、測試、程式碼三者綁在一起，這是 Rust 工程品質的一大亮點。

### 模組層級的文件 `//!`

還有一種 `//!`（驚嘆號），用來寫「**這個模組／這個 crate 整體**」的說明，通常放在檔案最上方：

```rust
//! # 我的數學工具庫
//!
//! 這個 crate 提供基本的數學運算函式。

/// 計算兩數之和。
pub fn add(a: i32, b: i32) -> i32 { a + b }
```

說明：`//!` 描述「包含它的那個東西」（整個檔案/模組），`///` 描述「下面那個項目」。兩者搭配，就能產生有總覽、有細節的完整文件。

## 小練習

1. 為 [rust-7-3] 的 `add` 函式寫文件註解（含一段 `# 範例`），跑 `cargo doc --open` 看產生的網頁。
2. 在範例裡故意寫一個**錯的** `assert_eq!`（例如說 `add(2,3)` 等於 `6`），跑 `cargo test`，看 doc test 抓出這個錯誤。
3. 在 `src/main.rs` 或 `lib.rs` 最上方用 `//!` 寫一段「這個專案是做什麼的」說明，重新產生文件看看。
