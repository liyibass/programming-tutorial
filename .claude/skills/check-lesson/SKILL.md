審查指定章節是否符合規範。

使用方式：`/check-lesson [檔案路徑]`
例如：`/check-lesson lessons/part-2/2-3-compound-types.md`

## 撰寫規範（最新版）

!`cat CLAUDE.md`

## 執行步驟

1. 讀取使用者指定的 `.md` 檔案
2. 判斷章節類型：主線課程（`lessons/`）或課外讀物（`課外讀物/`）
3. 逐項對照以下 checklist，標記 ✅ 通過 或 ❌ 未通過 或 ➖ 不適用

**基本結構 Checklist：**
- [ ] 有「本章目標」（一句話）
- [ ] 概念說明遵循「類比/pseudo code → 圖表 → 程式碼」順序
- [ ] 抽象概念有 pseudo code 或生活類比先行
- [ ] 有適當的 Mermaid 圖表（流程/結構/時序有需要的話）
- [ ] 程式碼範例前有說明文字
- [ ] 有「小練習」區塊（課外讀物章節標 ➖ 不適用）
- [ ] 術語第一次出現時有解釋
- [ ] 沒有連續超過 30 行未加說明的程式碼

**課外讀物引用 Checklist：**
- [ ] 如有對應課外讀物，使用標準 Markdown 連結格式（`> 想深入了解 XXX → [課外讀物 E-X-X：標題](../../課外讀物/...)`），不是純文字
- [ ] 如是 POC 解鎖章節，有 POC 引導區塊

**SOLID / Best Practice 引導 Checklist：**
- [ ] 程式碼涉及函式拆分 → 有引導至 E-7-2（SRP）或 E-6-3（若無此情境標 ➖）
- [ ] 程式碼涉及 TypeScript 介面設計 → 有引導至 E-7-5（ISP）（若無此情境標 ➖）
- [ ] 程式碼涉及模組依賴關係 → 有引導至 E-7-6（DIP）（若無此情境標 ➖）
- [ ] 出現反模式 → 有「常見錯誤」區塊並引導至 E-6-6（若無此情境標 ➖）
- [ ] 涉及 Git 操作 → 有引導至 E-8 對應章節（若無此情境標 ➖）
- [ ] 涉及測試概念 → 有引導至 E-9 對應章節（若無此情境標 ➖）

4. 輸出每項的通過狀態，❌ 項目附上具體說明（缺什麼、建議在哪裡補）
5. 最後總結：「可以發布 ✅」或「需要修改 N 項 ❌」
